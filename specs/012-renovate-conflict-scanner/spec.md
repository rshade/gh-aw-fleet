# Feature Specification: Renovate Config Conflict Scanner (Advisory)

**Feature Branch**: `012-renovate-conflict-scanner`
**Created**: 2026-06-14
**Status**: Draft
**Input**: GitHub issue #100 — "feat(security): detect Renovate configs that conflict with gh-aw-managed pins"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Warn when Renovate could bump the compiler-coupled action (Priority: P1)

As a fleet operator running `deploy` (or `sync`/`upgrade`) against a managed repo,
I want to be warned when that repo's Renovate configuration does **not** disable
updates to the `gh-aw-actions` packages, so that I can add the disable rule before
Renovate independently bumps the action and breaks the gh-aw integrity manifest /
hash validation.

**Why this priority**: This is the core motivator. The `gh-aw-actions` version is
coupled to the `gh aw` compiler version baked into lock-file metadata; the fleet
manages it atomically via `gh aw upgrade`. An out-of-band Renovate bump silently
breaks hash validation across the repo's agentic workflows. Catching this at
deploy time — before Renovate ever opens its first PR — is where the value lands.

**Independent Test**: Point the scanner at a clone whose Renovate config is present
but lacks the gh-aw-actions disable rule; confirm exactly one advisory finding is
produced, that it names the gap, and that it quotes a copy-pasteable rule block the
operator can drop into their config. Delivers value on its own even if no other
rule is ever checked.

**Acceptance Scenarios**:

1. **Given** a managed repo whose Renovate config exists but contains no rule
   disabling `gh-aw-actions` package updates, **When** the operator runs a deploy,
   **Then** the run surfaces one advisory finding describing the missing rule and
   including the exact rule block to add.
2. **Given** a managed repo whose Renovate config already disables `gh-aw-actions`
   package updates, **When** the operator runs a deploy, **Then** no
   gh-aw-actions-related finding is produced.
3. **Given** the finding above, **When** the operator reads it on any output
   surface, **Then** the remediation text is complete enough to copy-paste without
   further lookup.

---

### User Story 2 - Warn when Renovate could rewrite generated lock files (Priority: P2)

As a fleet operator, I want to be warned when a managed repo's Renovate
configuration does **not** exclude the generated `*.lock.yml` files from Renovate,
so that I can add the exclusion before Renovate rewrites pinned action SHAs /
container tags that gh-aw regenerates on every compile.

**Why this priority**: Lock files are regenerated by `gh aw compile`; any Renovate
edit to them is reverted on the next compile and can break the integrity manifest
in the meantime. High value, but secondary to the gh-aw-actions coupling because
the breakage is more self-correcting (next compile overwrites it).

**Independent Test**: Point the scanner at a clone whose Renovate config is present
but does not exclude the generated lock-file glob; confirm exactly one advisory
finding is produced quoting the exact exclusion block to add.

**Acceptance Scenarios**:

1. **Given** a managed repo whose Renovate config exists but does not exclude the
   generated lock-file glob, **When** the operator runs a deploy, **Then** the run
   surfaces one advisory finding describing the missing exclusion and including the
   exact rule block to add.
2. **Given** a managed repo whose Renovate config already excludes the generated
   lock-file glob, **When** the operator runs a deploy, **Then** no
   lock-file-related finding is produced.
3. **Given** a managed repo whose Renovate config lacks **both** rules, **When** the
   operator runs a deploy, **Then** two distinct advisory findings are produced (one
   per missing rule).

---

### User Story 3 - Stay silent and safe when there is nothing actionable (Priority: P3)

As a fleet operator, I want the scanner to make no noise and never block when a
managed repo has no Renovate config, an already-correct config, or a config that
cannot be parsed — so that adding this check never turns a working deploy into a
failed or noisy one.

**Why this priority**: Robustness and trust. An advisory scanner that false-alarms
on repos that don't even use Renovate, or that aborts a deploy on a malformed file,
would be worse than not having the feature. This slice guarantees the scanner is
"quiet by default" and strictly non-blocking.

**Independent Test**: Run the scanner against (a) a clone with no Renovate config,
(b) a clone with a correct config, and (c) a clone with a deliberately malformed
config; confirm (a) and (b) produce zero findings, (c) produces at most one
informational note, and none of the three blocks or fails the operation.

**Acceptance Scenarios**:

1. **Given** a managed repo with no Renovate config in any recognized location,
   **When** the operator runs a deploy, **Then** the scanner produces no finding and
   the deploy proceeds unchanged.
2. **Given** a managed repo whose Renovate config cannot be parsed, **When** the
   operator runs a deploy, **Then** the scanner produces a single informational
   finding noting the config could not be read, and the deploy still proceeds.
3. **Given** any Renovate finding at all, **When** it is produced, **Then** it never
   changes the success/failure outcome of the deploy/sync/upgrade operation.

### Edge Cases

- **Equivalent-but-different rule form**: a config that disables the same packages
  or files using a different but semantically equivalent expression than the
  canonical block. Per FR-012 (intent-based, equivalence-aware matching), this is
  treated as "rule present" and produces no finding.
- **Multiple candidate config files present**: a repo containing more than one
  recognized Renovate config location. The scanner inspects the first match in a
  fixed probe order and does not merge across files (see Assumptions).
- **Config inherits rules remotely**: a config whose rules come from a shared preset
  / `extends` reference rather than being inline. The scanner inspects only the
  local config file and does not resolve remote presets (see Assumptions); this can
  produce an advisory finding even when the effective config is correct.
- **Empty or rule-less config**: a present-but-minimal config with no relevant rules
  (and no repository-wide disable) yields both findings (treated identically to
  "rules missing").
- **Renovate disabled repo-wide**: a config whose top-level setting turns Renovate
  off for the entire repository makes any conflict impossible; the scanner treats
  both required rules as satisfied and produces no finding.
- **Comment / trailing-comma syntax**: a config using JSON5-style comments or
  trailing commas must be accepted, not treated as malformed.
- **Config present but not actually used by Renovate** (e.g. Renovate not enabled on
  the repo at all): out of the scanner's knowledge; it reports on file contents only.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST detect whether a managed repo's clone contains a Renovate
  configuration by probing a fixed, ordered set of recognized config locations
  (repo-root and `.github/` variants) and using the first one found.
- **FR-002**: System MUST accept Renovate configs written with JSON5-style comments
  and/or trailing commas, parsing them without treating that syntax as malformed.
- **FR-003**: When a Renovate config is present but does not disable Renovate updates
  to the `gh-aw-actions` package family, System MUST emit exactly one advisory
  finding that identifies the missing rule and includes the exact rule block the
  operator should add.
- **FR-004**: When a Renovate config is present but does not exclude the generated
  agentic-workflow lock files from Renovate — whether by an exact lock-file glob or a
  broader glob that covers them — System MUST emit exactly one advisory finding that
  identifies the missing exclusion and includes the exact rule block to add.
- **FR-005**: When no Renovate config is present in any recognized location, System
  MUST emit no finding for this scanner.
- **FR-006**: When a Renovate config is present but cannot be parsed, System MUST
  emit a single informational finding and MUST NOT panic, return an error, or block
  the operation.
- **FR-007**: Findings from this scanner MUST be advisory only — they MUST NOT block,
  fail, or otherwise change the success/failure outcome of a deploy, sync, or
  upgrade.
- **FR-008**: Findings MUST surface on every existing security-finding output
  surface: the operator's stderr stream, the structured JSON `warnings[]` envelope,
  and the pull request's security-findings section.
- **FR-009**: System MUST NOT modify the managed repo's Renovate config or any other
  file in the clone; the scanner is strictly read-only.
- **FR-010**: Each conflict finding MUST include remediation text complete enough to
  resolve the gap by copy-paste, without requiring the operator to consult external
  documentation.
- **FR-011**: System MUST emit each Renovate **conflict** finding (the missing
  gh-aw-actions rule and the missing lock-file exclusion) at the `LOW` advisory
  severity tier — a non-blocking level that ranks these actionable
  config-gap recommendations above purely informational notes but below
  blocking-grade findings. The malformed-config finding (FR-006) remains `INFO`,
  consistent with the existing convention for "could not read / parse" notices.
- **FR-012**: System MUST decide whether each required rule is "present" using an
  intent-based, equivalence-aware match: a rule is considered present if **any**
  configuration rule disables updates to the `gh-aw-actions` package family (matched
  by package name, regardless of which match key, wording, or block shape expresses
  it) — and, independently, if **any** configuration rule excludes the generated
  lock-file glob. Findings fire only when the *intent* is absent, not when the exact
  canonical block is absent, to avoid false positives on healthy configs that achieve
  the same disable with different syntax. Recognized equivalents include alternate
  rule expressions (different match keys or broader globs), repository-level ignore
  lists, and a repository-wide Renovate disable (which makes any conflict impossible).
- **FR-013**: The Renovate scanner MUST be registered alongside the existing security
  scanners and run in the same operations where security findings are already
  collected and surfaced (deploy, sync, and upgrade).
- **FR-014**: System MUST NOT introduce any new third-party dependency to deliver
  this scanner, and MUST NOT change the JSON output envelope's schema version (the
  new findings are additive within the existing warnings contract).

### Key Entities *(include if feature involves data)*

- **Renovate configuration**: the managed repo's Renovate config file, discovered by
  probing a fixed set of recognized locations. The unit of input the scanner reads.
- **Conflict rule**: one of two policies the fleet needs present in that config — (A)
  disable updates to the `gh-aw-actions` package family, (B) exclude the generated
  agentic-workflow lock files. Each missing rule maps to one finding.
- **Advisory finding**: a non-blocking record carrying the affected file, a
  human-readable description of the gap, and copy-pasteable remediation; flows
  through the existing finding pipeline to stderr, JSON warnings, and the PR body.
  Conflict findings carry `LOW` severity; the malformed-config note carries `INFO`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For a managed repo whose Renovate config is present but lacks both
  required rules, the scanner produces exactly two advisory findings — one per
  missing rule.
- **SC-002**: For a managed repo whose Renovate config already contains both required
  rules, the scanner produces zero findings.
- **SC-003**: For a managed repo with no Renovate config, the scanner produces zero
  findings and the deploy/sync/upgrade outcome is identical to a run without this
  feature.
- **SC-004**: 100% of conflict findings include a remediation block the operator can
  apply by copy-paste without external lookup.
- **SC-005**: No deploy, sync, or upgrade run is ever blocked or failed by this
  scanner across all tested inputs (present-correct, present-missing-each-rule,
  absent, malformed).
- **SC-006**: A malformed Renovate config never causes a crash or error exit; it
  yields at most one informational finding.
- **SC-007**: The change adds no new third-party dependency and leaves the JSON
  output envelope's schema version unchanged.

## Assumptions

- **First-match probe order**: when multiple recognized Renovate config locations
  exist in one clone, the scanner inspects the first match in a fixed probe order and
  does not merge or cross-check across files.
- **Local file only**: the scanner inspects the contents of the discovered config
  file only. It does not resolve shared/remote Renovate presets (`extends`),
  organization-level config, or a `renovate` key embedded in `package.json`; a repo
  whose rules are entirely inherited may therefore receive an advisory finding even
  though its effective config is correct.
- **Runs where security findings already run**: the scanner participates in the
  existing security-finding collection that occurs during deploy, sync, and upgrade,
  and uses those operations' existing output surfaces — no new command or flag is
  introduced.
- **Comment-tolerant parsing reuses existing capability**: tolerant parsing of
  comment/trailing-comma config is available through machinery the project already
  uses for its own config files; no new parsing dependency is required.
- **Fleet self-fix is parallel and optional**: adding the same two rules to the
  fleet's own repository's Renovate config is a worthwhile parallel cleanup but is
  not required for this scanner to deliver value and is not part of the scanner's
  acceptance.

## Out of Scope

- **Auto-patching** a managed repo's Renovate config. Deferred as an explicit opt-in
  follow-up; the project's comment-preserving write machinery covers only the fleet's
  own files, not managed repos' configs, and rewriting managed-repo files conflicts
  with the "thin orchestrator" value. This feature is detect-and-warn only.
- **Resolving remote/shared Renovate presets** to evaluate the effective config.
- **The Dependabot conflict-scanner variant** (sibling issue) — tracked separately;
  this spec covers Renovate only.
- **Verifying that Renovate is actually enabled** on the managed repo; the scanner
  reports on config file contents, not on the repo's Renovate activation state.
