# Feature Specification: Compile Workflows with --strict on Public Repos by Default

**Feature Branch**: `010-compile-strict-public`
**Created**: 2026-05-17
**Status**: Draft
**Input**: GitHub issue #49 — `feat(deploy/upgrade): compile workflows with --strict on public repos by default`

## Clarifications

### Session 2026-05-17

- Q: Should the effective compile-strict outcome surface as typed fields on `DeployResult` / `UpgradeResult` (visible in `--output json`), or is structured logging sufficient? → A: Option A — add typed fields. Matches the spec 005 precedent (`ActionsDisabled`, `WorkflowTokenReadOnly`) so the envelope contract stays consistent across preflight findings. Two new fields, no `cmd.SchemaVersion` bump (additive booleans default to `false`; consumers ignore unknown fields).
- Q: How should the fleet handle an operator's local `gh aw` CLI that predates the `--strict` flag (i.e., `< v0.68.3`)? → A: Option A — probe `gh aw compile --help` once per deploy/upgrade *only when* `EffectiveCompileStrict` returns `true`, detect absence of `--strict` in the help output, and abort cleanly *before* invoking compile (so the failure surfaces as a precondition error rather than a raw CLI stderr). Probe failures (e.g., `gh aw` itself missing) abort with a separate diagnostic naming the missing tool. Sets a new pattern (no version-probe elsewhere today) but bounded to this feature's call sites.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Public repo gets strict-compiled workflows automatically (Priority: P1)

A fleet operator runs `gh-aw-fleet deploy rshade/gh-aw-fleet --apply`. The target is a public GitHub repository. The default profile pulls in `agent-performance-analyzer`, whose upstream frontmatter declares `strict: false`. Today the resulting PR lands `.lock.yml` files compiled in non-strict mode; once merged, the workflow refuses to run on the public repo with a "Recompile the workflow with --strict to meet the security requirements for public repositories" error. The operator must then manually `gh aw compile --strict` and open a second PR. With this feature, the deploy detects that the repo is public, runs `gh aw compile --strict` in the work-dir clone after every `gh aw add` completes (before `git add .github/`), and the resulting PR contains strict-compiled lock files that pass the public-repo runtime check on first merge.

**Why this priority**: This is the entire reason the issue exists. Every other story in this spec exists to make this story safe to ship — without US1 working, the feature does nothing. P1 because it eliminates a recurring foot-gun (one repo + one strict-false workflow = one wasted PR cycle) that scales linearly with fleet size.

**Independent Test**: On a known-public test repo, configure a profile that includes `agent-performance-analyzer` (or any workflow shipping `strict: false`) and no `compile_strict` setting in either `fleet.json` or `fleet.local.json`. Run `gh-aw-fleet deploy <repo> --apply`. Inspect the resulting PR's `.lock.yml` files and confirm they were produced by `gh aw compile --strict` (e.g., absence of `permissions: write-all`, pinned action SHAs). Confirm the post-merge workflow run completes without the "Recompile with --strict" runtime warning.

**Acceptance Scenarios**:

1. **Given** a public repo with no `compile_strict` setting and a profile that includes at least one `strict: false` workflow, **When** the operator runs `gh-aw-fleet deploy <repo> --apply`, **Then** `gh aw compile --strict` is invoked exactly once in the clone directory after all `gh aw add` calls complete and before any `git add .github/` invocation, the resulting `.lock.yml` files are strict-compiled, and the operator sees a structured log line at info level naming the repo and resolution source (`auto-public`).
2. **Given** the same public repo, **When** the operator runs a dry-run `gh-aw-fleet deploy <repo>` (no `--apply`), **Then** the dry-run output indicates that strict compile would be applied (resolution: `auto-public`) without actually invoking the compile step, so the operator can confirm the policy before committing.
3. **Given** an `upgrade` invocation against the same public repo, **When** `gh-aw-fleet upgrade <repo> --apply` runs, **Then** the same compile step fires after `runUpgrade` succeeds, the resulting upgrade PR contains strict-compiled lock files, and the behavior is symmetric with US1's deploy path.

### User Story 2 - Explicit operator override beats auto-detection (Priority: P1)

The operator has two repos with non-default needs: one is public but legitimately can't pass strict compile (e.g., a legacy migration workflow that uses `permissions: write-all` by necessity for a one-shot data migration); one is private but the operator wants strict-mode hygiene anyway in anticipation of making it public. With this feature, the operator sets `"compile_strict": false` for the first repo in `fleet.local.json` (suppressing the auto-on-because-public default) and `"compile_strict": true` for the second (forcing the auto-off-because-private default). Both settings beat visibility auto-detection.

**Why this priority**: P1 alongside US1 because secure-by-default without an escape hatch is operator-hostile — operators who hit a legitimate edge case must be able to opt out without forking the tool. The "explicit override beats auto" rule is also the only thing that gives operators a way to *raise* strictness above the auto-detected baseline on private repos.

**Independent Test**: Two test repos — one public, one private. For each, set `compile_strict` explicitly to the opposite of the visibility default in `fleet.local.json` (`false` for public, `true` for private). Run `gh-aw-fleet deploy` on each. Assert: the public-repo deploy skips the compile step (override wins), and the private-repo deploy invokes the compile step (override wins again). Both log lines name the resolution source as `explicit` rather than `auto-public`/`auto-private`.

**Acceptance Scenarios**:

1. **Given** a public repo with `"compile_strict": false` in `fleet.local.json`, **When** the operator runs `gh-aw-fleet deploy <repo> --apply`, **Then** the compile step is skipped, an info-level log line names the repo and resolution source as `explicit` (value: `false`), and the resulting PR contains non-strict-compiled lock files.
2. **Given** a private repo with `"compile_strict": true` in `fleet.local.json`, **When** the operator runs `gh-aw-fleet deploy <repo> --apply`, **Then** the compile step is invoked, an info-level log line names the resolution source as `explicit` (value: `true`), and the resulting PR contains strict-compiled lock files.
3. **Given** a repo declared in both `fleet.json` (no `compile_strict` field) and `fleet.local.json` (with `"compile_strict": true`), **When** loaded, **Then** the loader's existing overlay semantics make the local file's setting the effective value — there is no special precedence rule introduced by this feature.

### User Story 3 - Fail-secure when visibility lookup fails (Priority: P2)

The operator runs deploy from a CI environment where the `GH_TOKEN` lacks the scope needed to read a repo's metadata, or the GitHub API is temporarily unavailable, or the repo's visibility flipped between two API calls in the same session. In every such case, the tool cannot determine whether the target is public. Rather than guess wrong, the tool defaults to strict ON and emits a `warn`-level log line naming the repo and the lookup failure reason. The operator's options are then explicit: set `compile_strict: false` to opt out, or fix the underlying credential/network issue and retry. The opposite default (strict OFF on lookup failure) would silently land a runtime-broken workflow on what was, in fact, a public repo — exactly the foot-gun this feature exists to prevent.

**Why this priority**: P2 because it's a non-regression / safety-rail requirement rather than added user value, but it is the gating condition that makes US1 and US2 safe — without fail-secure semantics, lookup flakiness becomes a new silent failure mode.

**Independent Test**: With a token that returns `403` from `/repos/<owner>/<repo>` (or with the API stubbed to return a 5xx), run `gh-aw-fleet deploy <repo>` against a repo that has no explicit `compile_strict` setting. Assert: (a) the deploy invokes the compile step (strict ON), (b) one `warn`-level structured log line names the repo and the lookup error, (c) the deploy still proceeds and the rest of the pipeline runs normally.

**Acceptance Scenarios**:

1. **Given** a repo with no explicit `compile_strict` setting and a `gh api /repos/<owner>/<repo>` call that returns a non-200 response (403, 404, 5xx, network error, or malformed JSON), **When** the deploy executes, **Then** the resolution falls back to strict ON, a single `warn`-level structured log line records the repo and the failure reason (HTTP status or error class), and the deploy proceeds with the compile step invoked.
2. **Given** the same lookup failure but the operator has set `"compile_strict": false` explicitly, **When** the deploy executes, **Then** the explicit override beats the fail-secure default (compile is skipped) and no `warn` log fires from this resolver (the explicit setting bypassed the lookup entirely).
3. **Given** the same lookup failure during an `upgrade` invocation, **When** the upgrade executes, **Then** behavior is symmetric with deploy — fail-secure ON, single `warn` log, pipeline proceeds.

### User Story 4 - Compile failure surfaces an actionable hint (Priority: P2)

The operator deploys a profile that includes a workflow which legitimately cannot be made strict — for instance, an upstream workflow that hardcodes `permissions: write-all` or uses an action that hasn't been pinned to a SHA. With this feature, `gh aw compile --strict` fails. The deploy aborts before `git add .github/`, leaving the work-dir clone preserved for inspection. The operator's terminal shows a hint generated by `diagnostics.CollectHints` that names the failing workflow (parsed from compile stderr), names the most likely cause (strict-mode violation detected), and surfaces the `compile_strict: false` opt-out path as one of the suggested remediations — so the operator has the right knob in hand without re-reading the docs.

**Why this priority**: P2 because operators hit this rarely (most workflows compile cleanly under strict), but when they do hit it, the difference between "raw stderr that says `compile failed: exit 1`" and "structured hint that names the workflow and the escape hatch" is the difference between a 5-minute fix and a half-hour rabbit hole. Reuses the existing diagnostic pattern from `internal/fleet/diagnostics.go`.

**Independent Test**: Construct a profile that includes one workflow with a known unfixable-strict violation (e.g., committed locally to the work-dir before compile runs). Run `gh-aw-fleet deploy <repo>` against a public repo. Assert: (a) the compile step runs, (b) it fails, (c) the deploy aborts with a non-zero exit, (d) the work-dir clone is preserved (per existing `--apply` failure semantics), (e) the printed error contains a `diagnostics.CollectHints` entry that names the workflow and mentions the `compile_strict: false` override path, not just raw compile stderr.

**Acceptance Scenarios**:

1. **Given** a deploy where the compile step fails because at least one workflow violates strict mode, **When** the deploy aborts, **Then** the error output includes an actionable hint that names the offending workflow (or at minimum, the strict-mode-failed error class) and mentions `"compile_strict": false` as one of the documented opt-out paths.
2. **Given** the same deploy under `--apply`, **When** it aborts, **Then** the work-dir clone is preserved (per the existing failure-preservation contract — `/tmp/gh-aw-fleet-*` is not cleaned), allowing the operator to inspect, fix manually, and resume with `--work-dir`.
3. **Given** a compile failure during `upgrade`, **When** it aborts, **Then** the same hint shape is surfaced (symmetric diagnostic between deploy and upgrade paths).

### User Story 5 - Onboarding feedback at `fleet add` time (Priority: P3)

The operator runs `gh-aw-fleet add <owner/repo>` to onboard a new target. As part of the add operation, the tool fetches the repo's visibility once and prints an info line naming the policy that will apply on the next deploy: e.g., `"public repo: workflows will be compiled with --strict on next deploy (auto-on; override with \"compile_strict\": false)"` for a public target, or the symmetric private-repo message. The operator learns the policy at the moment they're staging the repo into `fleet.local.json`, not later when they wonder why an unfamiliar compile step ran.

**Why this priority**: P3 because the operator can always discover the same information by running a dry-run deploy. But surfacing it at `add` time is good ergonomics — it removes the surprise factor for operators who don't know this feature exists.

**Independent Test**: Run `gh-aw-fleet add <public-repo>` and `gh-aw-fleet add <private-repo>` against two test targets. Assert: each invocation prints a single info line accurately describing the deploy-time compile policy and naming the override config key. The line appears in stdout (alongside the existing add-confirmation output), not stderr (it's expected behavior, not a warning).

**Acceptance Scenarios**:

1. **Given** the operator runs `gh-aw-fleet add <public-repo>`, **When** the add succeeds, **Then** one stdout info line names the auto-on policy and the override key (`compile_strict: false`).
2. **Given** the operator runs `gh-aw-fleet add <private-repo>`, **When** the add succeeds, **Then** one stdout info line names the auto-off policy and the inverse override key (`compile_strict: true`).
3. **Given** the visibility lookup fails (403/5xx/network error), **When** the add succeeds, **Then** the info line is omitted (the operator will learn the resolution at deploy time via the `warn` log from US3) and the add itself does NOT fail because of the lookup failure.

### Edge Cases

- **`internal` visibility**: GitHub repos can be `public`, `private`, or `internal` (enterprise org default). Internal repos are not subject to the public-repo strict-mode runtime check, so they MUST be treated as `private` for compile-strict purposes (auto-off when no explicit setting). Only the literal string `"public"` triggers auto-on.
- **Repo redirect (rename)**: `gh api /repos/<old>/<repo>` follows the GitHub redirect to the new canonical repo; the visibility returned belongs to the canonical repo. No special handling.
- **Repo archived**: Archived repos return the same `visibility` field shape as live repos. Auto-detection applies normally.
- **`gh aw compile` unavailable**: `gh aw` is invoked via the gh extension. The FR-016 probe runs `gh aw compile --help` before invoking compile and distinguishes two failure modes: (1) `gh aw` binary missing or extension not installed → probe exits non-zero → abort with "gh aw not found" diagnostic; (2) `gh aw` installed but predates `--strict` → probe succeeds but output lacks the flag → abort with "gh aw too old, minimum v0.68.3" diagnostic. Both cases preserve the work-dir clone and surface through `diagnostics.CollectHints`.
- **`gh aw` newer than v0.68.3 but `--strict` flag renamed/removed in a future version**: The FR-016 probe treats absence of the literal `--strict` token as version-too-old. If a future `gh aw` release renames the flag, operators will see the same diagnostic and must upgrade the spec to recognize the new flag name. This is acceptable — the spec versions with the upstream CLI rather than carrying compatibility shims.
- **Multi-workflow `runAdd` loop**: Compile runs once per repo on the entire `.github/workflows` directory (the natural `gh aw compile` invocation shape), not once per workflow. This means a single failing workflow aborts compile for all — which is the desired behavior (a public repo with one strict-fail workflow shouldn't ship the rest in strict mode while that one is silently dropped).
- **`extra-workflows` (per-`RepoSpec.ExtraWorkflows`)**: Extra workflows compile under the same repo-level setting as profile workflows; there is no per-workflow strict override (out of scope per issue).
- **`--work-dir` resume**: When deploy resumes from a prior interrupted run via `--work-dir`, the compile step MUST re-run (the prior run may have only completed `runAdd` and crashed before compile). The compile is idempotent — re-running it on already-strict-compiled lock files produces no diff.
- **Visibility flips mid-operation**: Visibility is resolved once per Deploy/Upgrade/Sync invocation (not per workflow). If a repo flips visibility between two invocations, the operator sees that on the next run. No caching across runs (per issue: "Visibility caching across multiple deploy invocations" is out of scope).
- **`fleet.json` overlay precedence**: The existing loader merges `fleet.json` base with `fleet.local.json` overlay. `compile_strict` follows the same overlay rule — a value in `fleet.local.json` shadows a value in `fleet.json`. No special precedence introduced.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `RepoSpec` MUST gain an optional tri-state `compile_strict` field (JSON name `compile_strict`, Go pointer-to-bool with `omitempty`) representing operator override of the auto-detected default: `nil` (auto-detect), `true` (force ON), `false` (force OFF).
- **FR-002**: The loader MUST accept `compile_strict` in either `fleet.json` or `fleet.local.json` and apply the existing overlay merge rule — `fleet.local.json` value shadows `fleet.json` value when both are present. Adding the field MUST NOT bump `fleet.SchemaVersion` (additive change, absent on existing files round-trips byte-identically).
- **FR-003**: The Config object MUST expose an `EffectiveCompileStrict(ctx, repo)` resolver that returns `(effective bool, source string, reason string)` with this resolution order: (1) if `RepoSpec.CompileStrict` is non-nil, return `(*spec.CompileStrict, "explicit", "")`; (2) otherwise, fetch repo visibility via `gh api /repos/<owner>/<repo>` once — `"public"` → `(true, "auto-public", "")`, any other value (including `"private"` and `"internal"`) → `(false, "auto-private", "")`; (3) when the API call fails (non-200, malformed response, network error, missing field), return `(true, "auto-fallback", truncate(err.Error(), 200))` so the caller fails-secure (strict ON) AND has the truncated reason available for the FR-007 warn log. The resolver never returns an error — fail-secure semantics fold the lookup error into the `auto-fallback` source + `reason` string.
- **FR-004**: Deploy MUST invoke `gh aw compile --strict` (no per-workflow arguments) in the work-dir clone exactly once, after all `runAdd` calls for the deploy complete and before `git add .github/`, but only when `EffectiveCompileStrict` returns `true`. When it returns `false`, the compile step MUST be skipped entirely.
- **FR-005**: Upgrade MUST invoke the same compile step under the same condition, after `runUpgrade` succeeds (and after `runUpdate` if also run) and before `git add .github/`. This is symmetric with FR-004; Sync inherits the compile step transitively because it delegates to Deploy via `applyDeployOrPrune` and does NOT introduce its own duplicate compile.
- **FR-006**: When the compile step is invoked, deploy/upgrade MUST emit one info-level structured log line naming the repo, the effective strict value (`true`), and the resolution source (one of: `explicit`, `auto-public`, `auto-fallback`). When the compile step is skipped, the same shape MUST be emitted with effective value `false` and resolution source (`explicit` or `auto-private`).
- **FR-007**: When the visibility lookup fails and the resolution falls back to strict ON, deploy/upgrade MUST emit ONE `warn`-level structured log line naming the repo and the failure reason (HTTP status code or error class). The `info`-level line from FR-006 still fires with resolution source `auto-fallback`. The deploy MUST proceed regardless.
- **FR-008**: When the explicit override (`CompileStrict != nil`) is set, the visibility-lookup call MUST be skipped entirely — no `gh api /repos/<owner>/<repo>` invocation. This avoids both unnecessary API quota use and the noise of fail-secure `warn` logs when the operator has already decided the policy.
- **FR-009**: When `gh aw compile --strict` itself fails (non-zero exit), deploy/upgrade MUST abort the operation, MUST preserve the work-dir clone (existing failure-preservation contract — clone dir at `/tmp/gh-aw-fleet-*` is not removed), MUST NOT proceed to `git add` / `git commit` / `git push` / PR creation, and MUST surface the compile failure through `diagnostics.CollectHints` — adding at least one new hint pattern that recognizes strict-mode validation failure output and points the operator at the `"compile_strict": false` opt-out path. Raw compile stderr MUST still be included in the error so the operator can see the underlying violation.
- **FR-010**: `gh-aw-fleet add <owner/repo>` MUST, on success, fetch repo visibility once and emit ONE info line on stdout describing the deploy-time policy that will apply, naming the override config key. When the visibility lookup fails, the info line MUST be omitted and the add itself MUST succeed (the operator will see the resolution at deploy time via the FR-007 `warn` log).
- **FR-011**: The compile step's stderr/stdout MUST be tee'd to the operator's terminal (or captured as a structured event for `--log-format json`), matching the pattern established by `runUpgrade`/`runUpdate` — operators always see what their tools just did.
- **FR-012**: Adding this feature MUST NOT introduce any new third-party dependencies (constitution Principle I). The implementation reuses the existing `exec.CommandContext` + `gh api` invocation pattern from `ghAPIExists` / `ghAPIJSON` for visibility lookup, and the existing `runAdd` / `runUpgrade` shape for the compile invocation.
- **FR-013**: The visibility lookup MUST be observably read-only — only `GET` against `/repos/<owner>/<repo>`. No `PATCH`/`PUT`/`POST`/`DELETE` calls introduced by this feature.
- **FR-014**: The implementation MUST inject the visibility fetcher behind a package-level `var func(...)` (precedent: `ghAPIExists`, `ghAPIJSON`, `ghDiscussionsAPI`, `ghRunArtifactAPI`) so unit tests run offline with fixture closures. No real network calls in `make test`.
- **FR-015**: `DeployResult` and `UpgradeResult` MUST gain two typed fields populated from `EffectiveCompileStrict`: `CompileStrictApplied bool` (JSON `compile_strict_applied`) and `CompileStrictSource string` (JSON `compile_strict_source`, one of `"explicit"`, `"auto-public"`, `"auto-private"`, or `"auto-fallback"`). Both fields MUST appear in the `--output json` envelope. The addition MUST NOT bump `cmd.SchemaVersion` — the boolean defaults to `false` (no-finding semantics) and the string defaults to the empty string (which envelope consumers MUST treat as "not applicable to this result," distinct from any of the four resolution-source values). The fields MUST be populated on every Deploy/Upgrade result that reached the resolver, including the resume-from-work-dir path that already populates `MissingSecret` / `ActionsDisabled` / `WorkflowTokenReadOnly`.
- **FR-016**: Before invoking `gh aw compile --strict`, deploy/upgrade MUST probe the local CLI by executing `gh aw compile --help` exactly once per invocation, parsing the output for the literal token `--strict`. The probe MUST run *only* when `EffectiveCompileStrict` resolves to `true` — when strict is off, the probe is skipped (no extra exec). When the probe output lacks `--strict`, the operation MUST abort *before* invoking compile, MUST preserve the work-dir clone, MUST NOT proceed to `git add` / `git commit` / `git push`, and MUST emit a structured error via `diagnostics.CollectHints` naming the minimum required `gh aw` version (`v0.68.3`) and the operator's currently-detected version (best-effort, parsed from `gh aw --version` output if available). When the probe itself fails (non-zero exit from `gh aw compile --help`, or `gh aw` binary not found), the operation MUST abort with a separate diagnostic distinguishing "gh aw missing / broken" from "gh aw too old."
- **FR-017**: The probe-output parser and the `gh aw --version` parser MUST both be injected behind package-level `var func(...)` seams (matching the FR-014 pattern for the visibility fetcher) so unit tests cover the four probe outcomes — flag present, flag absent, probe-exec-failed, version-parse-failed — without invoking real `gh aw` subprocesses.

### Key Entities

- **Compile-Strict Resolution**: A discriminated outcome `{effective: bool, source: "explicit" | "auto-public" | "auto-private" | "auto-fallback"}` produced by `EffectiveCompileStrict`. `auto-fallback` covers all lookup-failure paths (403, 404, 5xx, network error, malformed JSON, missing field) and is always paired with a `warn` log per FR-007.
- **RepoSpec.CompileStrict**: The operator-controlled tri-state knob (`*bool` with `omitempty`). The pointer-vs-value distinction is load-bearing — a plain bool cannot represent "not set" vs "explicitly false."
- **Compile Invocation**: One `gh aw compile --strict` call per Deploy/Upgrade run when effective is `true`, executed in the work-dir clone after all `runAdd`/`runUpgrade` activity completes and before any `git add` activity begins.
- **Strict-Mode Diagnostic Hint**: A new entry in `internal/fleet/diagnostics.go` that recognizes strict-mode validation failure patterns in compile stderr and emits a hint mentioning the workflow name (when parseable) and the `compile_strict: false` opt-out path.
- **DeployResult / UpgradeResult Extension**: Two new fields per the spec 005 precedent — `CompileStrictApplied bool` (`true` only when the compile step actually ran in this invocation) and `CompileStrictSource string` (one of the four discriminant values, empty when no resolver call occurred). Surfaced in the `--output json` envelope so CI consumers can gate on `.compile_strict_applied == false && .compile_strict_source == "auto-fallback"` — the actionable case where the deploy proceeded with strict ON because visibility lookup failed.
- **CLI Capability Probe**: A precondition check executed once per deploy/upgrade (only when strict is effective) that runs `gh aw compile --help` and looks for the `--strict` token. Discriminates three outcomes: `flag-present` (proceed), `flag-absent` (abort with a version-too-old diagnostic), `probe-failed` (abort with a gh-aw-missing diagnostic). The probe is the operator's first signal that their local CLI is incompatible — surfaced *before* any actual compile attempt so the error is a clean precondition failure rather than a noisy CLI stderr dump.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For a public-repo deploy with no explicit `compile_strict` setting and at least one upstream `strict: false` workflow in the profile, the resulting PR contains `.lock.yml` files that are strict-compiled — verifiable by deploying to a known-public test repo, opening the PR, and asserting that running the workflow post-merge does NOT produce the "Recompile with --strict" runtime warning.
- **SC-002**: For a public-repo deploy with `"compile_strict": false` explicitly set, the resulting PR's lock files are NOT strict-compiled — verifiable by capturing the lock-file content before and after toggling the override and asserting a diff.
- **SC-003**: For a private-repo deploy with `"compile_strict": true` explicitly set, the resulting PR's lock files ARE strict-compiled — verifiable by inspecting the PR content against a known-private test repo.
- **SC-004**: When the visibility lookup fails (token scope insufficient, network error, etc.), the deploy completes successfully with strict ON applied AND emits exactly one `warn`-level structured log line — verifiable by stubbing the visibility fetcher to error and asserting one log entry plus successful deploy completion.
- **SC-005**: When `gh aw compile --strict` fails, the deploy aborts before any push activity, the work-dir clone remains on disk, and the printed error contains a hint mentioning either the failing workflow name or the `compile_strict: false` opt-out path — verifiable by injecting a fixture compile failure and asserting (a) no PR is created, (b) the clone dir still exists, (c) `diagnostics.CollectHints` output appears in the error message.
- **SC-006**: Unit tests cover `EffectiveCompileStrict` across all five resolution branches (explicit-true, explicit-false, nil+public, nil+private, nil+lookup-error) PLUS the `internal` visibility edge case (six rows total — `internal` exercises the same `auto-private` branch as `private` but locks in the edge-case mapping), via fixture closures injected through the package-level visibility-fetcher `var func(...)` — no real network calls, total `make test` runtime growth under 200ms.
- **SC-007**: Adding `"compile_strict"` to an existing `fleet.local.json` produces a byte-identical round-trip on save (via the HuJson AST-mutation write path), and removing it from a `RepoSpec` likewise round-trips — verifiable by parsing a sample file, adding/removing the field, serializing, and diffing.
- **SC-008**: `make ci` passes (gofmt, vet, golangci-lint, full test suite) with no new lint suppressions added in `.golangci.yml`. Any new exported identifier carries a godoc comment per the CLAUDE.md "Code self-documentation" contract.
- **SC-009**: A `--output json` envelope captured from a deploy invocation contains both `compile_strict_applied` and `compile_strict_source` fields whose values match the FR-006 log line for the same invocation — verifiable by extracting both via `jq` and asserting equality with the log stream's structured event for the resolver. Empty-string and `false` defaults must round-trip unchanged on results that never reached the resolver.
- **SC-010**: Unit tests cover the FR-016 probe across all three outcomes (flag-present, flag-absent, probe-exec-failed) via fixture closures injected through the FR-017 package-level `var func(...)` seams — no real `gh aw` invocations. When the probe outcome is `flag-absent`, the test asserts (a) the operation aborts before `git add` / commit / push activity, (b) the work-dir clone is preserved, (c) the error message names both the minimum required version (`v0.68.3`) and the operator's detected version (when parseable from `gh aw --version`).

## Assumptions

- **Visibility is fetched once per Deploy/Upgrade invocation** (not cached across invocations, not cached across multi-repo iterations within a single invocation if/when the tool grows to multi-repo). Caching is explicitly out of scope per the source issue.
- **`internal` visibility is treated as private** for compile-strict purposes. Only the literal `"public"` string from the `visibility` field triggers auto-on. This matches GitHub's runtime check, which is scoped to public repos.
- **Sync does not need its own compile step**: Sync's `applyDeployOrPrune` path delegates to Deploy, which already invokes compile under FR-004. Sync's drift-only / dry-run paths do not produce new `.lock.yml` files, so no compile is needed there.
- **The fleet's internal Layer 1 security scanner (`#38`) is independent of this feature.** This feature governs only the `gh aw compile --strict` pass-through (a different "strict" concept). If `#38` lands first with a generic `--strict` CLI flag, the issue author has flagged that it should be renamed to `--scan-strict` before merging to preserve naming separation. This spec does not block on `#38` and `#38` does not block on this spec.
- **No CLI flag for compile-strict in this slice**: configuration-only via `compile_strict` in `fleet.json` / `fleet.local.json`. If a one-off override becomes operator-painful, a future issue can add `--compile-strict` / `--no-compile-strict` flags (deliberately distinct from `#38`'s future `--scan-strict`).
- **No per-workflow strict override**: granularity is repo, not workflow. An operator who needs different strictness per workflow within the same repo can `exclude` the workflow from the profile and re-add it via `ExtraWorkflows` with its own settings (though within the same repo, the compile setting still applies uniformly).
- **The `gh aw compile --strict` invocation does NOT rewrite workflow `.md` files** — it overrides frontmatter at compile time only, producing strict-compiled `.lock.yml` files. The "Never rewrite workflow markdown" hard invariant from CLAUDE.md is preserved.
- **The HuJson round-trip semantics for `fleet.local.json` already handle additive optional fields cleanly**: adding `compile_strict: false` via direct AST mutation or RFC 6902 patch preserves operator comments and other fields without re-serializing the whole file (precedent: `Add` and `SaveTemplates` in `internal/fleet/load.go`).
- **No `cmd.SchemaVersion` bump is required.** Resolved by the 2026-05-17 clarification (Q1 → Option A): the two new fields are additive (boolean defaults to `false`, string defaults to `""`), JSON consumers ignore unknown fields, and the spec 005 precedent established this as non-breaking. The wire-contract surface grows but does not change for existing consumers.
- **`gh aw >= v0.68.3` is required only when compile-strict is effective.** Resolved by the 2026-05-17 clarification (Q2 → Option A): the probe runs only when `EffectiveCompileStrict` returns `true`, so operators running entirely on private repos with no explicit `compile_strict: true` overrides will not incur the version requirement. The README MUST document this conditional requirement so onboarding operators know when their local CLI version matters.
- **The probe sets a new pattern, scoped to this feature.** No retroactive version-checking for `runAdd` / `runUpgrade` / `runUpdate` / fetch is implied — those invocations continue to trust the local CLI as before. If subsequent features need their own version probes, they should follow the FR-016/FR-017 injection pattern rather than building a global version-gate.
