# Feature Specification: Fleet Overview Subcommand

**Feature Branch**: `018-overview-subcommand`
**Created**: 2026-06-23
**Status**: Draft
**Input**: User description: "Add a new read-only `gh-aw-fleet overview` subcommand that renders one consolidated cross-repo dashboard joining run health (failures / success rate), cost (AI credits and derived USD), and drift (desired vs actual workflow refs) into a single per-repo row plus a fleet total. Ships free, runs locally, no SaaS/auth/tenancy. Reuses the drift path the status command already has and the run-log fan-out the consumption command already has — no duplicated API logic. Default temporal window is the trailing 7 days; exit code is non-zero only on drift (run failures are advisory); cost honors the existing nil-until-positive rule. Tracks GitHub issue #153; validates demand for the hosted control-plane epic (#146) as the cheap local wedge that precedes it."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - One-command fleet health, cost, and drift dashboard (Priority: P1)

A fleet operator running agentic workflows across ten or more repositories wants one local command that answers "is my fleet healthy, what is it costing me, and which repos have drifted?" — without running two separate commands and then bouncing to each repo's Actions tab in a browser. Today drift lives in one command, cost lives in another, and run health (are the workflows actually *passing*?) is surfaced nowhere in the tool. The new subcommand fans out across the fleet and joins all three signals into a single row per repo: drift state, run counts with a success percentage, and consumption in AI credits with a derived dollar figure, finished off with a fleet-wide total row. The operator reads the whole posture of the fleet in one screen.

**Why this priority**: This is the entire reason the feature exists. Even with no temporal flags, no repo scoping, and no machine-readable output, a single joined dashboard that puts drift, health, and cost on one line per repo is the minimum viable product — it eliminates the "run two commands and check ten Actions tabs" toil that motivated the request. Every later capability refines this one view.

**Independent Test**: Point the tool at a small fleet of two or three repositories that have agentic workflows installed, run the subcommand with no flags, and confirm a single table appears with one row per repo carrying a drift state, a run count, a failure count, a health percentage, and consumption figures, closed by a total row that aggregates every repo. The command must render the table from one invocation — the operator never has to run a second command or open a browser to assemble the same picture.

**Acceptance Scenarios**:

1. **Given** a fleet of three repos that all have agentic workflows and recent runs, **When** the operator runs the overview subcommand with no flags, **Then** the output displays exactly three repo rows — each showing drift state, total runs, failed runs, a success percentage, and consumption (AI credits plus derived USD) — followed by one total row aggregating runs, failures, a pooled fleet success percentage, and summed consumption.
2. **Given** a fleet where one repo has drifted workflows, one is aligned, and one cannot be reached, **When** the operator runs the overview subcommand, **Then** each repo's row shows its respective drift state (`drifted`, `aligned`, `errored`) in the drift column without the operator needing to run a separate drift command.
3. **Given** a fleet of three repos with mixed run outcomes, **When** the operator runs the overview subcommand, **Then** the health column for each repo reflects that repo's own success rate over the window and the cost columns reflect that repo's own consumption, so per-repo hotspots and failure clusters are both visible at a glance.

---

### User Story 2 - Recency window and repo scoping (Priority: P2)

After landing the single-screen dashboard, the same operator wants to ask the question the way they actually think about health: "is my fleet failing *lately*?" Run health is inherently a windowed concept — a single most-recent snapshot is the wrong default for "is anything breaking this week." The subcommand therefore windows health and cost over a trailing period by default, and offers the same three mutually exclusive temporal selectors the consumption command already exposes: a trailing-window mode, a since-a-fixed-date mode, and a single-most-recent mode. The operator can also name one or more specific repos to narrow the dashboard to just those, for focused triage of a subset of the fleet.

**Why this priority**: Once the P1 joined view exists, recency and scoping are the next natural refinements every operator reaches for. P2 because the default-window dashboard already delivers the core value; windowing and scoping make it answer sharper questions but are additive on the same fan-out.

**Independent Test**: With a fleet that has been running workflows for over a week, invoke the subcommand once with the trailing-window mode set to seven days and again with a since-date mode set to a date three days ago, and confirm the run/failure/cost figures cover the expected span in each mode. Then invoke it naming a single repo and confirm only that repo's row (and the total) renders. Confirm that passing more than one temporal selector is rejected.

**Acceptance Scenarios**:

1. **Given** any fleet, **When** the operator runs the overview subcommand with no temporal flag, **Then** the dashboard covers the trailing seven days by default, and the output makes the active window visible so the operator knows what span the numbers describe.
2. **Given** a fleet that has been running workflows for two weeks, **When** the operator runs the subcommand with the trailing-window mode set to seven days, **Then** the run, failure, and cost figures reflect only runs within the last seven days.
3. **Given** any fleet, **When** the operator passes more than one of the trailing, since-date, and most-recent temporal selectors in a single invocation, **Then** the command exits with a non-zero status and a clear message stating the selectors are mutually exclusive.
4. **Given** a fleet of five repos, **When** the operator names two of them as arguments, **Then** the dashboard renders rows for only those two repos plus a total row scoped to them, and the other three repos are excluded entirely.
5. **Given** the operator names a repository that is not declared in the fleet configuration, **When** they run the subcommand, **Then** the command fails fast with a clear message identifying the unknown repository rather than silently rendering an empty or partial dashboard.

---

### User Story 3 - CI-safe drift gate with advisory run health (Priority: P2)

An operator wants to wire the overview into a continuous-integration check that fails the pipeline when the fleet has drifted from its declared desired state — the same gate semantics the standalone drift command already provides — while *not* flapping the pipeline every time an upstream agentic workflow has a flaky run. Drift is a property of the fleet's own desired-state contract and is the operator's to fix; run failures reflect upstream workflow execution the operator does not control from this tool. So the command's exit status keys on drift only: non-zero when any repo is drifted or errored, zero otherwise — even when the health column shows failures. Run failures remain fully visible in the output as advisory signal; they just do not change the exit code.

**Why this priority**: This is what lets the overview double as an automated guardrail, not just a human dashboard, which is a meaningful step up in value. P2 because the human-facing dashboard (Stories 1 and 2) is already useful on its own; CI-gating is an additive contract on top of it.

**Independent Test**: Construct a fleet where every repo is aligned but at least one repo has failing runs, run the subcommand, and confirm the process exits zero (failures are advisory). Then construct a fleet with at least one drifted repo and confirm the process exits non-zero. Confirm a fleet that is fully aligned with all-passing runs exits zero.

**Acceptance Scenarios**:

1. **Given** a fleet where every repo is aligned but one repo's runs are all failing, **When** the operator runs the overview subcommand, **Then** the dashboard shows the failures in the health column and the process exits with a zero (success) status — run failures are advisory.
2. **Given** a fleet where at least one repo is drifted, **When** the operator runs the overview subcommand, **Then** the process exits with a non-zero status.
3. **Given** a fleet where at least one repo is errored (could not be queried for drift), **When** the operator runs the overview subcommand, **Then** the process exits with a non-zero status, matching the drift command's treatment of unreachable repos.
4. **Given** a fleet that is fully aligned with all runs passing, **When** the operator runs the overview subcommand, **Then** the process exits with a zero status.

---

### User Story 4 - Machine-readable output for automation (Priority: P3)

An operator building dashboards, alerts, or pipeline steps on top of the fleet tool wants the overview as structured data, not just a rendered table, so downstream tooling can consume the joined health/cost/drift signal without scraping columns out of human-formatted text. The subcommand therefore offers the same structured output envelope every other read-only fleet command already honors, carrying the per-repo joined rows, the fleet totals, and any diagnostics, under the existing envelope contract so downstream consumers need no new schema knowledge.

**Why this priority**: Structured output unlocks programmatic consumers and is the bridge to any future hosted/automated layer, but the human-facing dashboard already delivers the headline value. P3 because it is a parallel rendering of an already-complete result, not a new capability.

**Independent Test**: Run the subcommand with structured output selected against a fixture fleet and confirm the emitted envelope validates against the project's existing envelope shape, contains one structured entry per repo with the joined drift/health/cost fields, contains the fleet totals, and carries the same diagnostics that the text output would surface — all without bumping the envelope's schema version.

**Acceptance Scenarios**:

1. **Given** any fleet, **When** the operator runs the overview subcommand with structured output selected, **Then** the command emits the project's standard output envelope containing per-repo joined entries, a fleet-total entry, and a diagnostics collection, and the envelope's schema version is unchanged from prior releases.
2. **Given** a fleet where one repo is inaccessible, **When** the operator runs the subcommand with structured output, **Then** that repo's entry is present and marked as errored, and the corresponding diagnostic appears in the envelope's diagnostics collection rather than aborting the emission.
3. **Given** a repo whose only runs in the window failed, **When** the operator runs the subcommand with structured output, **Then** that repo's consumption fields are absent (not zero) in the envelope, honoring the nil-until-positive convention.

### User Story 5 - Spot run-rate waste from no-op runs (Priority: P2)

In a real deployed fleet, most scheduled agentic runs are no-ops: they complete successfully but decide "no action needed," while still consuming AI credits to reach that decision. A fleet that is overwhelmingly no-op reads as nearly 100% healthy and yet quietly burns budget producing nothing — and that waste is invisible in a plain pass/fail/cost view. The operator wants the dashboard to make it explicit: a no-op count per repo, sitting next to runs, failures, and health, so "we are paying for a lot of runs that do nothing" jumps off the row instead of hiding behind a healthy-looking success rate. This is the run-rate-efficiency lens — the same question the fleet's own audit workflow tries to answer, but computed locally and for free, and available even when that workflow is down.

**Why this priority**: Once the joined dashboard exists, the no-op breakdown is the dimension that turns "healthy and cheap-looking" into an honest cost-efficiency picture — it is where the operator decides which scheduled workflows to slow down or drop. P2 because the basic join (Story 1) is already actionable, and the no-op column is an additive signal layered onto it from the same data.

**Independent Test**: Point the tool at a fleet where one repo's runs in the window are overwhelmingly no-ops, run the subcommand, and confirm the no-op column reflects that the repo's runs were mostly no-action, the health figure stays high (no-ops are successes), and the cost column is non-blank (no-ops still cost credits) — so the operator can see the waste even though nothing "failed."

**Acceptance Scenarios**:

1. **Given** a repo whose window contains 4 productive successes, 36 no-op successes, and 2 failures, **When** the operator runs the overview subcommand, **Then** the row shows RUNS 42, FAIL 2, NOOP 36, and HEALTH 95% — no-ops are counted as healthy successes and surfaced as a distinct column, with the cost columns non-blank because the no-op successes consumed credits.
2. **Given** a repo whose every run in the window was a no-op success, **When** the operator runs the overview subcommand, **Then** the row shows RUNS equal to NOOP, zero failures, a 100% health figure, and non-blank cost columns — the canonical run-rate-waste signal: healthy, but paying for runs that did nothing.
3. **Given** any fleet, **When** the operator runs the overview subcommand, **Then** the fleet-total row includes a summed no-op count so fleet-wide run-rate waste is readable at a glance.

### Edge Cases

- A repo declared in the fleet configuration has agentic workflows but had zero runs within the selected window (freshly onboarded, or simply idle). The row still renders with zero runs and zero failures; the health figure is shown as not-applicable (a distinguishable placeholder) rather than as either 0% or 100%, because a success rate over zero runs is undefined. Cost columns render as absent under the nil-until-positive rule.
- A repo declared in the fleet configuration has no agentic workflows at all (nothing compiled to a lock file). The drift signal still computes; the health and cost columns render as not-applicable. The repo is not dropped from the dashboard.
- A repo's only runs in the window all failed. Run and failure counts are non-zero, the health figure is 0%, and the cost columns render as absent — failed runs do not report consumption, so under the nil-until-positive rule the credits and dollar figures are blank rather than zero.
- A repo's entire window is no-op successes (the common steady-state for a quiet, fully-scheduled fleet). RUNS and NOOP are equal and non-zero, FAIL is zero, HEALTH is 100%, and the cost columns are non-blank because no-op successes still consume credits. This row is the canonical run-rate-waste signal — distinct from a genuinely idle repo (zero runs, blank cost) — and lets the operator tell "paying for nothing" apart from "doing nothing."
- The fleet's own audit / consumption agentic workflow (`audit-workflows`, `api-consumption-report`) is undeployed, stale, or failing on a repo. The overview MUST still render that repo's run-rate, health, and cost from raw run-log data, because it computes them independently and does not read those workflows' Discussions output. Observed in the live fleet: the audit workflow had gone silent on one repo and had never posted on another, yet the per-repo run telemetry remained fetchable.
- A repo is inaccessible for the drift query (permissions, a deleted or renamed repo, a transient network error). That repo's drift cell is marked errored and its row still renders with whatever health and cost data could be fetched; the failure is surfaced as a per-repo diagnostic and the rest of the fleet renders normally. One bad repo never blanks the whole dashboard.
- A repo is reachable for drift but its run-log fan-out fails (or vice versa). Each of the two signals degrades independently: the cell that could not be fetched is marked unavailable while the cell that succeeded still renders, and a diagnostic identifies which signal failed for which repo.
- The platform reports the operator is rate-limited partway through the fan-out. Already-fetched repos render their full row; repos not yet fetched surface a rate-limit diagnostic rather than the whole command aborting with a hard error.
- The fleet configuration declares zero repos, or the operator scopes to a set that resolves to zero repos. The command exits successfully with an empty dashboard and a top-level diagnostic explaining the fleet is empty, distinct from a crash.
- A run in the window has a non-terminal conclusion (still in progress or queued at fetch time). It is excluded from both the run total and the failure count for the window, so an in-flight run never inflates the failure rate; the totals describe completed runs only.
- A run in the window was `cancelled` (e.g. by a concurrency rule on a rapid push) or `skipped`. Per the failures-only health taxonomy it is excluded from both the run total and the failure count — it is neither a success nor a failure — so a burst of concurrency cancellations does not depress a repo's health figure. A repo whose every run in the window was cancelled or skipped therefore reads like a zero-completed-runs repo: RUNS 0, a not-applicable health placeholder, and blank cost columns.
- The since-date temporal mode is set beyond the platform's run-log retention window for one of the repos. The row renders with whatever runs remain fetchable rather than erroring — a span truncated by retention reads the same as a quiet window (fewer or zero health-counting runs). Distinguishing retention-truncation from genuine inactivity would require parsing the upstream `gh aw logs` output for a retention signal and is deferred (research.md §9); this slice does not synthesize a dedicated retention diagnostic.
- Drift reports a repo as drifted but the same repo has perfectly healthy, low-cost runs. The exit code is still non-zero (drift gates the exit), while the health and cost columns truthfully show the repo is otherwise fine — the operator sees both facts on one row.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST expose a new read-only subcommand named `overview` that renders a single consolidated cross-repo dashboard joining drift, run health, and cost into one row per fleet repo.
- **FR-002**: The subcommand MUST NOT mutate any local or remote state — no pushes, no commits, no writes to any configuration file. It is a pure read command in the same category as the fleet listing, status, and consumption commands.
- **FR-003**: The subcommand MUST source its drift signal from the same drift-detection path the standalone status command already uses, and MUST NOT re-implement drift detection independently.
- **FR-004**: The subcommand MUST source its run-health and cost signals from the same run-log fan-out the consumption command already uses, and MUST NOT re-implement that fan-out or its documented quirks (the display-name-versus-slug requirement and the throwaway-working-directory artifact behavior).
- **FR-005**: For each repo in scope, the system MUST join the three signals into one row carrying at minimum: the repo identifier, the drift state, the health-counting run total in the window (RUNS, per FR-008), the failed-run count, the no-op-run count (NOOP, per FR-028), a success-rate figure, the consumption in AI credits, and the derived USD figure.
- **FR-006**: The drift cell MUST present one of the three drift states the status command defines — aligned, drifted, or errored.
- **FR-007**: The success-rate figure MUST be derived from a failures-only taxonomy: a run whose conclusion is `success` counts as healthy; a run whose conclusion is `failure`, `timed_out`, or `startup_failure` counts as a failure. Runs whose conclusion is `cancelled` or `skipped` MUST be excluded from the health calculation entirely — neither healthy nor failed — because agentic workflows are routinely cancelled by concurrency rules on rapid pushes, and counting those against health would make a healthy fleet read as broken.
- **FR-008**: The runs FR-007 excludes from the health calculation, together with any non-terminal conclusion (still in progress or queued at fetch time), MUST be excluded from both the run total and the failure count for the window. The reported run count (RUNS) therefore equals the number of runs that count toward health (successes plus FR-007 failures), so the displayed health rate is exactly successes ÷ RUNS and never penalizes a healthy fleet for routine cancellations or in-flight work.
- **FR-009**: The cost columns MUST honor the existing nil-until-positive convention: a credit total that is absent or non-positive MUST render as a blank placeholder (not zero), and the derived USD figure MUST be computed from the credit total using the project's existing fixed credit-to-USD rate.
- **FR-010**: A repo whose only runs in the window failed MUST render a 0% health figure with blank (absent) cost columns, because failed runs do not report consumption.
- **FR-011**: A repo with zero health-counting runs in the window (no successes and no FR-007 failures — whether because it was idle, had only `cancelled`/`skipped`/in-flight runs, or has no agentic workflows at all) MUST render with zero runs, zero failures, a not-applicable health placeholder (distinct from 0% and from 100%), and blank cost columns.
- **FR-012**: The text output MUST close with a fleet-total row aggregating runs, failures, no-op runs, a pooled fleet success rate (computed from pooled run and failure totals, not an average of per-repo rates), and summed consumption across the rows in scope.
- **FR-013**: The subcommand MUST support three mutually exclusive temporal selectors — a trailing-window mode, a since-a-fixed-date mode, and a single-most-recent mode — reusing the same selectors the consumption command exposes.
- **FR-014**: The default temporal window, when no selector is supplied, MUST be the trailing seven days — deliberately different from the consumption command's single-most-recent default, because health is a windowed concept. The active window MUST be visible in the output.
- **FR-015**: The system MUST reject invocations that combine more than one temporal selector, exiting with a non-zero status and a clear message naming the three modes.
- **FR-016**: The subcommand MUST accept zero or more positional repository arguments that scope the dashboard to only the named repos (plus a scoped total), reusing the fleet's existing repo-scoping behavior; with no arguments it covers the whole fleet.
- **FR-017**: When a named repository argument is not declared in the fleet configuration, the command MUST fail fast with a clear message identifying the unknown repository rather than rendering an empty or partial dashboard.
- **FR-018**: The command's exit status MUST be non-zero if and only if at least one in-scope repo is drifted or errored; it MUST be zero when every in-scope repo is aligned, regardless of run failures.
- **FR-019**: Run failures MUST be advisory: they MUST be fully visible in the output but MUST NOT change the exit status in this feature. (A future opt-in flag to gate on run failures is explicitly out of scope here.)
- **FR-020**: A repo that fails for one signal (drift or run-log fetch) MUST still render its row with whatever the other signal returned; the failing signal's cell MUST be marked unavailable/errored and the failure MUST surface as a per-repo diagnostic. A single failing repo MUST NOT blank the whole dashboard.
- **FR-021**: The subcommand MUST support both the default human-readable text output and the project's existing structured-envelope output that other read-only fleet subcommands honor. The structured envelope's schema version MUST NOT change — the new result type and its fields are additive.
- **FR-022**: The structured envelope MUST carry the per-repo joined entries, the fleet-total entry, and the diagnostics collection, so downstream consumers can read the joined signal without scraping the text columns.
- **FR-023**: The text output MUST surface a per-repo detail block for any repo that is drifted, errored, or otherwise unhealthy, mirroring the detail presentation the status command already uses, so the operator can see *what* drifted or *which* signal failed without a second command.
- **FR-024**: All operator-facing diagnostics — warnings, hints, rate-limit and inaccessible-repo notices — MUST be routed through the project's existing structured-diagnostics mechanism so they appear in both the text output and the structured envelope, and MUST reuse existing diagnostic codes (inaccessible repo, rate-limited, empty fleet) rather than introducing new ones unless a genuinely new join-time condition arises.
- **FR-025**: The system MUST NOT introduce any new third-party runtime dependency to satisfy this feature; it MUST rely on the project's existing libraries, standard-library tooling, and shell-out seams.
- **FR-026**: The system MUST NOT cache, persist, or baseline any overview result between invocations, and MUST NOT write any state outside the existing fleet configuration files. Each invocation re-discovers and re-fetches, preserving the fleet's stateless read model.
- **FR-027**: Operator-facing documentation — the project README, the agent-guidance files, and the subcommand's own usage text — MUST describe the command, the trailing-window default and its rationale, the drift-only exit-code contract, the advisory nature of run failures, the no-op / run-rate column and what it means, and the wall-clock cost of the run-log fan-out.
- **FR-028**: For each repo in scope, the system MUST surface a no-op-run count (NOOP) as a distinct column alongside RUNS, FAIL, and HEALTH. A no-op run is a run that completed successfully but took no action — the upstream `safe-outputs: noop` outcome (e.g. "no action needed," "nothing to process"). The column exists so the operator can see how many runs consumed credits while producing nothing — the run-rate-efficiency lens.
- **FR-029**: No-op runs MUST count as healthy successes in the HEALTH figure — they have a `success` conclusion and are not failures, consistent with the upstream framing that no-ops are successful outcomes. The NOOP count is therefore an informational subset of the successful runs (NOOP ≤ successes ≤ RUNS), never a separate failure bucket; HEALTH remains successes-including-no-ops over health-counting runs.
- **FR-030**: A no-op run's consumption (AI credits) MUST be carried into the cost columns like any other successful run — no-op successes still consume credits, so a no-op-heavy repo MUST show a non-blank cost. This is the signal that distinguishes "paying for runs that do nothing" from a genuinely idle repo.
- **FR-031**: The no-op count MUST be derived from the same `gh aw logs --json` fan-out used for health and cost, with no separate per-repo API path. The signal is the aggregate MCP tool-usage count for the `safeoutputs`/`noop` tool (the `mcp_tool_usage` summary's `call_count`), summed per repo across the workflow calls. This was empirically validated against the live fleet: for one workflow over a 30-day window the `noop` call count exactly matched the number of successful runs (9), confirming a no-op run emits one `noop` safe-output. The pinned gh-aw version exposes no per-run no-op flag (neither `safe_items_count`, which is unpopulated, nor `classification`, which is unrelated), so the NOOP column is a per-repo count, not per-run attribution, and the system MUST clamp it so it never exceeds that repo's successful-run count.
- **FR-032**: The overview's run-rate, health, and cost signals MUST be computed directly from run-log data and MUST NOT depend on any in-repo agentic workflow (such as `audit-workflows` / the Agentic Workflow Audit Agent, or `api-consumption-report`) being deployed, healthy, or posting to Discussions. The dashboard MUST remain fully available when those workflows are absent or failing — which is precisely when the operator most needs run-rate and health visibility.

### Key Entities *(include if feature involves data)*

- **Repo Overview Row**: The joined per-repo record — one row of the dashboard. Carries the repo identifier, the drift state (aligned / drifted / errored) plus enough drift detail to populate the detail block, the health-counting run count (successes plus FR-007 failures; `cancelled`/`skipped`/in-progress runs are excluded), the failed-run count, and the no-op-run count (an informational subset of the successes) over the window, the derived success rate (or a not-applicable marker when there were zero health-counting runs), the consumption in AI credits, and the derived USD figure (both absent under the nil-until-positive rule). Also carries per-signal availability so a row can report "drift errored but health/cost fetched" or vice versa.
- **Overview Result**: The top-level output container. Records the active temporal window, the set of joined per-repo rows, the fleet-total row, the diagnostics collection, and an overall drift-derived disposition that drives the exit code. Honors the project's existing output-envelope shape so downstream tooling needs no new schema knowledge.
- **Fleet Total Row**: The aggregate footer. Sums runs, failures, no-op runs, and consumption across the in-scope rows, and reports a pooled fleet success rate computed from the pooled run and failure totals rather than averaging per-repo rates. Consumption honors the same all-or-nothing nil-until-positive aggregation the consumption rollup uses.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A fleet operator can read drift, run health, and cost for every fleet repo from a single command invocation, replacing the prior workflow of running two separate commands and visiting each repo's Actions tab. For a ten-repo fleet, the operator-facing assembly time drops from several minutes of multi-command, multi-tab effort to one screen of output.
- **SC-002**: An operator can wire the overview into a CI check that fails on fleet drift without the check flapping on flaky upstream runs: a fleet that is fully aligned exits zero even when individual workflow runs failed, and a fleet with any drifted or errored repo exits non-zero.
- **SC-003**: Repos that cannot contribute a complete row — inaccessible, rate-limited, or with no data in the window — are identified by name in operator-visible diagnostics and still appear in the dashboard, so the operator can distinguish "no data" from "genuinely zero / genuinely healthy" and one bad repo never blanks the view.
- **SC-004**: The structured output envelope's schema version is unchanged after this feature ships; downstream consumers of the envelope need no schema update to read overview output, because the new result type is additive.
- **SC-005**: The feature adds no new third-party runtime dependency and persists no state between invocations; the dashboard is reproducible purely from live reads against the existing fleet configuration.
- **SC-006**: The full quality gate (formatting, vetting, linting, and the complete test suite as exercised by the project's CI target) passes cleanly with the new subcommand and its fixtures included, and no test in the new code requires live network access — every external call is exercised through the project's existing shell-out seams against captured fixtures.
- **SC-007**: When exactly one repo in an N-repo fleet is inaccessible, the operator still receives N−1 fully rendered rows plus the inaccessible repo surfaced as an errored row and a diagnostic — demonstrating per-repo failure isolation.
- **SC-008**: An operator can tell "paying for runs that do nothing" apart from "doing nothing" directly from the dashboard: a repo whose runs are predominantly no-ops shows a high health figure, a high no-op count, and non-blank cost, while a genuinely idle repo shows zero runs and blank cost — and this reading holds even when the repo's own observability workflows (the audit / consumption agentic workflows) are undeployed or failing, because the overview computes the numbers locally from run logs rather than reading those workflows' output.

## Assumptions

- The run-log fan-out reused for health and cost requires a recent-enough upstream agentic-workflow CLI that emits the AI-credit schema, the same minimum the consumption command already enforces. Older CLIs that decode without credit data are rejected by that shared gate; the overview inherits this behavior rather than re-checking it.
- Run health and cost are computed over the *same* temporal window in a given invocation. The window applies uniformly to both signals so the health and cost columns describe the same span; drift is a point-in-time desired-state comparison and is not windowed.
- A run's consumption (AI credits) is reported only for successfully completed runs; failed runs report no credits. This is why a repo whose only runs failed rolls up to blank cost columns — it is the existing, structural behavior of the upstream run-log data, not a special case invented here.
- The health rate uses a **failures-only taxonomy** (resolved during specification): `success` is healthy; `failure`, `timed_out`, and `startup_failure` are failures; `cancelled`, `skipped`, and non-terminal runs are excluded from the calculation entirely. This was chosen over a strict "anything-not-success is a failure" rule specifically because agentic workflows are routinely cancelled by concurrency rules on rapid pushes, and counting routine cancellations against health would make a healthy fleet read as broken. The displayed RUNS count is the health-counting denominator, so `HEALTH = successes ÷ RUNS` holds exactly.
- The default trailing-seven-day window is the right default for a health-oriented view; operators who want a single most-recent snapshot or a fixed-date span opt in via the temporal selectors. Seven days is a documented default, not a tunable global setting.
- Drift gates the exit code and run failures do not, because drift is a property of the fleet's own declared desired state (the operator's to fix) while run failures reflect upstream workflow execution outside this tool's control. An opt-in flag to also gate on run failures is a documented follow-up and is out of scope here.
- The run-log fan-out is the slow path; its wall-clock cost is inherited, not re-solved, in this feature. Performance improvements tracked separately (bounded concurrency / no-download fast path, and workflow-discovery pagination) are consumed when available rather than reimplemented. The expected wall-clock cost is documented in the command's usage text.
- Repos that participate in multiple profiles do not change the overview's per-repo model: the dashboard is per-repo first, so a repo appears exactly once regardless of how many profiles it carries. Profile / cost-center / workflow grouping axes are out of scope for this feature and can follow the consumption command's lead later.
- Operator-facing diagnostics route through the project's existing structured-diagnostics mechanism; the existing categories (inaccessible repo, rate-limited, empty fleet, generic hint) are expected to cover the surface area, and no new diagnostic code is introduced unless a genuinely new join-time condition emerges.
- No-op runs are a first-class, intended outcome of agentic workflows (the `safe-outputs: noop` feature) and are the dominant outcome in a quiet, fully-scheduled fleet. They are successes, not failures, so they count as healthy; they are surfaced as a distinct NOOP column because they still consume credits — and not trivially: a single observed no-op run (Sub-Issue Closer deciding "nothing to do") cost ~313 AI credits (~$3), so a steady stream of no-ops is real, recurring spend for zero output. The cost-efficiency story is otherwise invisible behind a healthy-looking success rate.
- The no-op count is sourced from the aggregate `safeoutputs`/`noop` MCP tool-usage count in `gh aw logs --json`, not from per-run records (the pinned gh-aw version exposes no per-run no-op flag, and `safe_items_count` is unpopulated). Because that aggregate is scoped to the `gh aw logs` fetch window (`--start-date`) rather than each run's timestamp, the NOOP column reflects the fetch window and is not client-side re-filterable per run the way RUNS / FAIL / AIC are. For the default trailing window these align (the fetch window is the reporting window); exact per-window no-op counts, if ever required, would come from the repo's `[aw] No-Op Runs` issue feed (marker `<!-- gh-aw-noop-runs -->`). The plan reconciles this minor windowing asymmetry (e.g. by clamping NOOP to the successful-run count) and validates the one-noop-call-per-no-op-run assumption.
- A fleet's own agentic workflows already attempt this measurement: `audit-workflows` (the "Agentic Workflow Audit Agent") computes per-workflow run rates, 24-hour and 30-day success/failure rates, token usage, and no-op tool-call counts, and `api-consumption-report` computes cost — both as scheduled LLM workflows that post to Discussions. The overview deliberately recomputes these signals locally from run logs instead of consuming those workflows' output, because (a) those workflows cost credits to run, (b) they are deployed unevenly across the fleet (observed live: present on some repos, absent on others), and (c) being agentic workflows themselves, they fail during the same outages they are meant to report on (observed live: one repo's audit feed went silent for ~2 weeks while its workflows were failing). Computing locally makes the overview free, uniform across the fleet, and available exactly when the in-repo observability workflows are down.

## Out of Scope

- The hosted control plane, authentication, multi-tenancy, and any web UI — this feature is the free local wedge that validates demand for that epic, not the epic itself.
- New grouping axes (by profile, by cost center, by workflow). The overview is per-repo first; grouping can follow the consumption command's lead in a later slice.
- Failing the command on run failures. Run failures are advisory in this slice; an opt-in flag to gate on them is a documented follow-up.
- Trend lines, history, alerting, or notifications.
- Per-failure cause breakdown (why a given workflow run failed) — that is a separate drill-down companion.
- Any caching, baseline, or persisted state outside the existing fleet configuration files.
- Consuming the output of the fleet's own agentic observability workflows (`audit-workflows`, `api-consumption-report`) as an overview data source — the overview computes its signals independently from run logs and does not depend on those workflows being deployed or posting. Reconciling the overview's local numbers against those workflows' Discussions reports is a possible later cross-check, not part of this slice.
- Deep no-op analytics beyond a per-repo and fleet-total count (e.g. per-workflow no-op rates, no-op trend over time, or recommending which scheduled workflows to slow down). The NOOP column surfaces the signal; acting on it is left to the operator or a later slice.
