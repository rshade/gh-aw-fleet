# Operator Quickstart: Fleet Manifest

**Feature**: `011-fleet-manifest`
**Audience**: Fleet operators upgrading to a release that includes this feature.

## What changed

Each `deploy --apply` and `sync --apply` now writes a fleet-owned manifest file into the managed
repo at `.github/aw/fleet-manifest.json`. The manifest records which version of `github/gh-aw`
was used to deploy, the CLI version, active profiles, and a timestamp.

`status` now reports a **version drift** dimension for each repo: `current`, `behind`, or
`unmanaged`.

## No action required to adopt

The manifest is created automatically on the next `deploy --apply` or `sync --apply`. Repos
without a manifest appear as `unmanaged` in `status` until their first post-feature deploy ŌĆö this
is expected and informational only.

## What you will see after the first post-feature deploy

The deploy PR for each repo will contain two new files:
- `.github/aw/fleet-manifest.json` ŌĆö the manifest (new file)
- Possibly refreshed init artifacts (`.github/agents/`, `.github/mcp.json`, etc.) if the repo
  was initialized by an older `gh aw` version

The init artifact refresh happens because the new `ensureInit` logic re-runs `gh aw init` when
no manifest is present (the "first time" case). This is safe ŌĆö init is idempotent for
`gh aw v0.79.2+`.

## Checking version drift

```sh
# Text output ŌĆö look for VERSION_DRIFT column
gh-aw-fleet status

# JSON ŌĆö machine-parseable
gh-aw-fleet status --output json | jq '.result.repos[] | {repo: .repo, drift: .version_drift}'

# Find repos that are behind
gh-aw-fleet status --output json \
  | jq '[.result.repos[] | select(.version_drift.state == "behind")]'
```

## What "behind" means

A repo is `behind` when its manifest records an older `github/gh-aw` source pin than the fleet
currently declares. This means the repo's init artifacts (skills, `.github/mcp.json`, agent
files) and lock files were generated by an older version of `gh aw`.

To refresh a `behind` repo:
```sh
gh-aw-fleet deploy owner/repo          # dry-run shows what will change
gh-aw-fleet deploy owner/repo --apply  # creates a PR with updated init artifacts + manifest
```

## Manifest file location

The manifest lives alongside `gh aw`'s own `actions-lock.json`:
```
.github/aw/fleet-manifest.json
```

This file is committed to the managed repo's history and is safe to read or inspect. It is owned
by the fleet tool ŌĆö do not hand-edit it.

## Same-version redeployes

Redeploying a repo at the same `github/gh-aw` version as the manifest records produces no manifest
diff. The timestamp is not updated, and the file is not touched. This means manifest-only noise is
not introduced into PRs by routine redeployes.

## Questions and troubleshooting

**"All my repos show `unmanaged`"** ŌĆö Normal after upgrading. Run `deploy --apply` or `sync
--apply` for each repo; the next PR will include the manifest.

**"A repo shows `behind` after I updated the fleet pin"** ŌĆö Also normal. Run deploy or sync
with `--apply` to refresh the init artifacts and write the new manifest.

**"Init failed on a legacy-layout repo"** ŌĆö If `gh aw init` fails during a deploy because the
repo was initialized by a very old CLI (pre-v0.68 dispatcher layout), the clone is preserved at
`/tmp/gh-aw-fleet-*`. Check the error, update the clone manually if needed, then re-run with
`--work-dir /tmp/gh-aw-fleet-<repo>`.
